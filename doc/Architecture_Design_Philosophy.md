## 1. 设计背景与问题分析

### 1.1 当前AI研究助手的痛点

作为AI领域的高级研究员，在使用现有AI研究工具时面临以下核心问题：

#### 问题1: 单一模型的局限性

```
现状: 大多数工具只使用单一AI模型
      ↓
问题: 不同任务需要不同模型的优势
      - 数学推理 → 需要o3的强逻辑能力
      - 中文处理 → 需要Qwen/DeepSeek的本地化优势
      - 实时信息 → 需要Grok的社交媒体访问
      - 多模态 → 需要Gemini的视觉理解
      ↓
结果: 单一模型无法在所有任务上表现最优
```

#### 问题2: 上下文窗口限制

```
现状: 研究过程产生大量中间结果
      ↓
问题: 直接塞入上下文会导致:
      - Token爆炸 (成本高)
      - 信息稀释 (重要信息被淹没)
      - 无法追溯 (人类难以复盘)
      ↓
结果: 复杂研究任务难以持续推进
```

#### 问题3: 缺乏"人文视角"

```
现状: 现有工具只关注技术本身
      ↓
问题: 研究不仅是技术，更是人的活动
      - 谁在推动这个方向？
      - 这个想法从哪里来？
      - 学派之间有什么竞争？
      ↓
结果: 缺失重要的研究历史和人物脉络信息
```

#### 问题4: 缺乏技术演进视角

```
现状: 只能获取当前技术状态
      ↓
问题: 不了解技术的来龙去脉
      - 这个技术为什么会出现？
      - 经历了哪些关键分叉？
      - 沿着什么轴在演进？
      ↓
结果: 难以把握技术发展规律，预测未来方向
```

#### 问题5: 输出质量不可控

```
现状: 直接使用模型原始输出
      ↓
问题: 缺乏质量保障机制
      - 提示词质量参差不齐
      - 上下文噪声大
      - 没有预处理和审查
      ↓
结果: 输出质量不稳定，可靠性存疑
```

### 1.2 设计目标

基于上述问题分析，NEXEN的设计目标明确为：

| 目标 | 解决的问题 | 核心方案 |
|------|-----------|---------|
| **异构多模型** | 单一模型局限 | 任务-模型最优匹配 |
| **外化记忆** | 上下文爆炸 | 分层存储，按需检索 |
| **人物谱系** | 缺乏人文视角 | Genealogist Agent |
| **技术历史** | 缺乏演进视角 | Historian Agent |
| **三模块流水线** | 输出质量不可控 | 提示词优化+记忆检索+预处理 |

### 2.2 设计原则

#### 原则1: 专业分工 (Specialization)

每个Agent只做自己最擅长的事，使用最适合的模型：

```python
SPECIALIZATION_MATRIX = {
    "逻辑推理": {"agent": "Logician", "model": "o3", "why": "o3在数学推理上SOTA"},
    "批判分析": {"agent": "Critic", "model": "o3-mini", "why": "快速、批判性强"},
    "人物追踪": {"agent": "Genealogist", "model": "Claude Opus", "why": "需要复杂推理和综合"},
    "中文处理": {"agent": "CN_Specialist", "model": "Qwen", "why": "中文处理最优"},
    "实时信息": {"agent": "Social_Scout", "model": "Grok", "why": "唯一能访问X/Twitter"},
    "多模态": {"agent": "Vision_Analyst", "model": "Gemini", "why": "多模态能力最强"},
}
```

#### 原则2: 记忆外化 (Memory Externalization)

不在上下文中堆积信息，而是写入文件系统，按需检索：

```
传统方式:
┌─────────────────────────────────────────┐
│  Context = 原始任务 + Agent1输出 +      │
│            Agent2输出 + Agent3输出 +    │
│            ... (无限增长)               │
│                                         │
│  问题: Token爆炸，信息稀释，无法追溯     │
└─────────────────────────────────────────┘

NEXEN方式:
┌─────────────────────────────────────────┐
│  Context = 原始任务 +                   │
│            精选Insights (L2) +          │
│            相关Digest (L1, 语义检索) +  │
│            Raw引用 (L0, 按需)           │
│                                         │
│  优势: Token可控，信息精炼，完全可追溯   │
└─────────────────────────────────────────┘
```

每个Agent执行前都要经过三模块流水线的质量保障：

```

│                      三模块质量闭环                              │

┌──────────┐     ┌──────────┐     ┌──────────┐

│(Gemini)  │     │(Claude)  │     │(Gemini)  │

              迭代直到通过

┌──────────┐     ┌──────────┐     ┌──────────┐

│(Gemini)  │     │ L2→L1→L0 │     │          │
```

## 3. 架构创新点详解

### 3.1 创新点1: 异构多模型协作

**传统方案**: 单一模型处理所有任务

**NEXEN方案**: 任务-模型最优匹配

```
┌─────────────────────────────────────────────────────────────────┐
│                    Model Router 智能路由                        │
└─────────────────────────────────────────────────────────────────┘

任务类型              最优模型           备选模型          原因
───────────────────────────────────────────────────────────────────
数学推理              o3                Claude Opus       o3逻辑能力最强
代码生成              Claude Sonnet     DeepSeek-Coder    Claude代码质量高
图表分析              Gemini 2 Pro      GPT-4V            Gemini多模态最强
中文内容              Qwen-Max          DeepSeek          中文训练数据充足
社交媒体              Grok 3            -                 唯一能访问X
复杂规划              Claude Opus       GPT-4o            长程推理能力强
提示词生成            Gemini 3 Pro      Claude            Gemini在此任务优化
```

**技术实现**:

```python
class ModelRouter:
    """智能模型路由器"""

    def route(self, task: Task) -> str:
        # 1. 语言路由
        if task.language == "zh" and task.type != "math":
            return "qwen-max"

        # 2. 模态路由
        if task.has_images:
            return "gemini-2-pro"

        # 3. 任务类型路由
        routing_table = {
            "math_reasoning": "openai/o3",
            "code_generation": "claude-sonnet",
            "social_media": "grok-3",
            "complex_planning": "claude-opus",
        }
        return routing_table.get(task.type, "claude-sonnet")
```

### 3.2 创新点2: 三层外化记忆系统

**传统方案**: 所有信息塞入上下文

**NEXEN方案**: 分层存储，按需检索

```
┌─────────────────────────────────────────────────────────────────┐
│                    三层记忆金字塔                                │
└─────────────────────────────────────────────────────────────────┘

                         ┌───────────┐
                         │    L2     │  ← 精华层 (必读)
                         │ insights/ │     ~2K tokens
                         │           │     关键发现、待解决问题
                      ┌──┴───────────┴──┐
                      │       L1        │  ← 摘要层 (语义检索)
                      │    digest/      │     ~4K tokens
                      │                 │     Agent摘要、主题摘要
                   ┌──┴─────────────────┴──┐
                   │          L0           │  ← 原始层 (按需引用)
                   │        raw/           │     不直接加载
                   │                       │     思维链、工具输出
                   └───────────────────────┘

检索策略:
1. 必读L2 (全部加载) ─────────────► ~2000 tokens
2. 语义检索L1 (top-k) ─────────────► ~4000 tokens
3. L0仅提供文件路径引用 ───────────► ~200 tokens
                                    ──────────────
                                    总计 ~6200 tokens (可控)
```

**优势对比**:

| 维度 | 传统方案 | NEXEN方案 |
|------|---------|----------|
| Token消耗 | 指数增长 | 线性可控 |
| 信息质量 | 噪声堆积 | 精炼提纯 |
| 检索效率 | 无法检索 | 语义搜索 |
| 可追溯性 | 不可追溯 | 完全可追溯 |
| 人类可读 | 不可读 | 分层可读 |
| 跨会话复用 | 无法复用 | knowledge_base持久化 |

## 7. 总结与展望

### 7.1 设计思路总结

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    NEXEN 设计思路核心                                        │
└─────────────────────────────────────────────────────────────────────────────┘

核心洞察:
  研究是复杂的人类活动，需要:
  • 不同专业的协作 → 异构多模型 + 多Agent
  • 长期记忆和追溯 → 外化记忆系统
  • 人文视角 → Genealogist
  • 历史视角 → Historian
  • 质量保障 → 三模块流水线

设计原则:
  • 专业分工: 让最强的模型做最擅长的事
  • 记忆外化: 避免上下文爆炸，保持可追溯
  • 质量闭环: 多重保障，确保输出质量
  • 人文+技术: 全方位理解研究问题

实现路径:
  • 渐进式: MVP → 增强 → 高级 → 优化
  • 模块化: 每个组件独立可替换
  • 可扩展: 新Agent、新技能易于添加
```
